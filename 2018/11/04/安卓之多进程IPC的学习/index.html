安卓之多进程IPC的学习

发表于 2018-11-04 | 分类于 Android
字数统计: 3.8k | 阅读时长 ≈ 15

1.简介

1.1 定义

IPC是Inter-Process-Communication的缩写，意思是进程间通信或者跨进程通信.

1.2 使用原因

  1. 静态成员和单例模式完全失效
  2. 线程同步机制完全失效
  3. SharedPreferences的可靠性下降
  4. Application多次创建

2.android的IPC方式

2.1 Bundle

四大组件中的三大组件(Activity,Service,Receiver)都是支持在Intent中传递Bundle数据的,由于Bundle实现了Parcelable接口,所以他可以方便在不同进程间传输,注意:我们在Bundle中的数据必须可以被序列化,比如基本数据类型,实现了Parcelable接口的对象,实现了Serializable接口的对象.

2.2 使用文件共享

1.使用机理

两个进程通过读/写同一个文件来交换数据.交换信息除了文本信息外,还可以序列化对象到文件在从另一个进程中读取这个对象,但是有一点需要注意,Android基于Linux,并发读/写文件没有限制,当两个线程同时写文件的时候可能会出现问题.

2.使用举例

3.使用局限性

当出现并发读写时候,读取的文件可能不是最新的.尽量避免这种情况或者使用线程同步来限制并发.SharedPreferences是个特例,它通过键值对方式存储数据,在底层上采用xml文件来存储键值对,本质上来说SharedPreferences也是属于文件的一种,但是由于系统对它的读写有一定的缓存策略,所以内存中会有一份SharedPreferences缓存,而在多进程模式下,系统对他读写变得不可靠,因此,在多进程通讯的时候最好不要使用SharedPreferences.
因此文件共享方式适合在对数据同步要求不高的进程间进行通信,并且需要妥善处理并发问题.

2.3 使用Messenger

1.使用机理

通过它可以在不同进程中传递Message对象,在Message中放 入我们想传递的数据,就可以实现数据的进程间传递了.Messenger底层实现是AIDL.由于它一次只能处理一个请求,因此在服务端不存在并发的问题.

2.具体使用

  • 服务端进程
    首先在服务端创建一个Service来处理客户端的连接请求,同时创建一个Handle并通过它来创建一个Messenger对象,然后在Service中的onBind中返回这个Messenger对象底层的Binder.
  • 客户端进程
    首先需要绑定服务端service,绑定成功后用服务端返回的IBinder对象创建一个Messenger,通过这个Messenger向服务端发送消息,发送的消息类型为Message对象,如果需要服务端能够回应客户端,就必须和服务端一样,创建一个Handle并创建一个新的Messenger,并把这个Messenger对象通过Message的replyTo参数传递给服务端,服务端就可以通过replyTo参数回应客户端.

2.1 客户端只向服务端发送消息

Messenger中进行数据传输必须将数据放入Message中,而Messenger和Message都实现了Parcelable接口,因此可以跨进程传输.Message中能作为载体传送数据的只有what,arg1,arg2,obj,replyTo,而obj在同一进程中是很实用的,但是进程间通讯的时候,在Android2.2以前obj不支持跨进程传递,2.2以后仅仅支持系统实现的Parcelable接口的对象才能通过它来传递,也就等于我们自定义的类即使实现了parcelable也无法通过obj传递.

  • 服务端可回应客户端消息

3.局限性

  1. Messenger是用串行的方式处理客户端发来的信息,因此,不能适用于有大量的并发请求发送到服务端的情况.
  2. Messenger的作用主要是为了传递信息,如果客户端需要调用服务端的方法的话,就不适用了.

2.4 使用AIDL

1. 具体使用

1.1 服务端

首先创建一个Service来监听客户端的连接请求,然后创建一个AIDL文件,将要给客户端的接口在AIDL文件中声明,然后在Service实现AIDL文件生成的类.最后在onBind方法返回实现的类.

1.2 客户端

首先绑定服务端的Service,将连接成功后返回的Binder对象转换成AIDL接口所属类,接着就可以调用AIDL中的方法了

1.3 AIDL接口的创建

1.4 AIDL中可以使用的类型

  • 基本数据类型(int,long,char,boolean,double等)
  • String 和 CharSequence;
    • List 中的所有元素都必须是以上列表中支持的数据类型、其他 AIDL 生成的接口或您声明的可打包类型。 可使用List泛型（例如，List< String >）。另一端实际接收的具体类始终是 ArrayList，生成的方法中支持的类型是 List 接口。
  • Map 中的所有元素都必须是以上列表中支持的数据类型、其他 AIDL 生成的接口或您声明的可打包类型。 不支持Map泛型（如 Map< String,Integer > 形式的 Map）。 另一端实际接收的具体类始终是 HashMap，生成的方法中支持的类型是 Map 接口
  • Parcelable: 所有实现了Parcelable接口的对象
  • AIDL: 所有的AIDL接口本身也是可以在AIDL文件中使用的

    注意:
    1.自定义的Parcelable对象和AIDL对象必须要显示的import进来,不管他们是否和当前AIDL文件位于同一包内.
    2. ,如果AIDL文件中使用了自定义的类,那么它必须继承Parcelable,因为Android系统可通过它将对象分解成可编组到各进程的原语.并且必须新建一个和它同名的AIDL文件,并在其中声明它为Parcelable类型.

1.5 远程服务端的实现

  • 创建一个Binder对象并在onBind中返回它,这个对象继承自IBookManager.Stub并实现了它内部的AIDL方法.
  • 采用了CopyOnWriteArrayList,这个CopyOnWriteArrayList支持并发读/写.AIDL方法是在服务的Binder线程池中执行的,因此在多个客户端同时连接的时候,会存在多个线程同时访问的情况,所以我们要在AIDL中处理线程同步,而我们这里直接使用CopyOnWriteArrayList来进行自动的线程同步.
  • AIDL中所支持的是抽象的List接口,因此虽然服务端返回的是CopyOnWriteArrayList,但在Binder中会按照List的规范去访问数据并最终形成一个新的ArrayList传递给客户端.

1.6 客户端的实现

  • 在客户端中绑定服务,绑定成功后将服务端返回的Binder对象转换成AIDL接口,然后就可以通过这个接口调用服务端的远程方法了.

2.结合观察者模式的使用

2.1 使用场景

用户不想时不时的查询图书列表了,于是,他去问图书馆能不能有新书直接告诉我.这个时候应该能马上想到,这是一种典型的观察者模式,每个感兴趣的用户都观察新书,当新书到的时候,图书馆就通知每个对这本书感兴趣的用户.

2.2 使用方法

  • 定义一个AIDL接口,因为AIDL无法使用普通接口,因此新建的是AIDL接口,该接口的作用是当服务端需要通知客户端的时候,可以调用该接口的方法,并把新书的对象传递给客户端.

  • 在之前的接口中添加两个方法,用来添加和删除监听者

  • 在服务端中定义一个存放监听器的容器,在添加和删除两个方法中添加和删除监听者.

  • 在服务端中定义一个添加书籍的方法,在该方法中添加书籍,并通知监听者

  • 在onCreate方法中开启线程,在线程中调用上面的onNewBookArrived方法.

  • 在客户端需要注册OnNewBookArrivedListener到远程服务端,同时要在Acyivity退出时解除注册,回调的onNewBookArrrived方法是在客户端的Binder线程池执行的,因此需要Handler切换到客户端的主线程中去执行.

2.3 解除注册

2.3.1 问题

当活动销毁的时候,我们解除注册到服务端的listener,发现解除注册的时候出现找不到我们之前注册的那个listener.

2.3.2 原因

Binder会把客户端传递过来的对象重新转化并生成一个新的对象,对象是不能跨进程直接传输的,对象的跨进程传输的本质上是反序列的过程.

2.3.3 解决方式

使用RemoteCallbackList是系统专门提供的用于删除跨进程listener的接口.它是一个泛型,支持管理任意AIDL接口.

2.3.4 RemoteCallbackList
  • 其内部有一个Map结构专门用来保存所有的AIDL回调,这个Map的key是IBinder类型,value是Callback类型.其中**Callback中封装了真正的远程listener.
  • 跨进程传输的对象底层的Binder对象是不变的,当客户解除注册的时候,可以遍历服务端所有的listener,找出和解注册listener具有相同Binder对象的服务端listener,并把它删掉.这是RemoteCallbackList的原理
  • 当客户端进程终止的时候,他能自动移除客户端所注册的listener
  • 内部自动实现线程同步的功能.
  • 使用时候(它并不是个list)

3.AIDL中的定向Tag

  • In类型的表现形式是：数据只能由客户端传向服务端，服务端对数据的修改不会影响到客户端
  • Out类型的表现形式是：数据只能有服务端传向客户端，即使客户端向方法接口传入了一个对象，该对象中的属性也是为空的，即不包含任何数据，服务端获取该对象后，对该对象的任何操作，就会同步到客户端这边，所以服务端在获取到客户端以Out方式传来的数据时会为空。
  • InOut类型，服务器对数据的改变也同步到客户端，因此可以说两者之间是双向流动的。

    4.AIDL中注意点

  • 客户端调用远程服务的方法,被调用的方法运行在服务端的BInder线程池,此时客户端的线程会挂起.此时客户端线程是UI线程,且服务端的方法比较耗时的时候,就会出现ANR.
  • 客户端的onServiceConnected和onServiceDisconnected方法运行在UI线程,因此不能调用非常耗时的服务端方法.
  • 服务端的方法本身就运行在服务端的BInder线程池中,因此服务端方法本身就可以执行大量的耗时操作,不要在服务端中开线程去进行异步任务.
  • 当远程服务端调用客户端的listener的方法时候,被调用的方法也运行在Binder线程池中,不过是客户端的线程池.因此如果服务端调用listener的方法的那个方法是UI线程中运行,它就不能在服务端中调用客户端的耗时方法.
  • 客户端的监听器的方法运行在客户端的Binder线程池中,因此不能在里面去访问UI的内容.
