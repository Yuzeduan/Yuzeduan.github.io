设计模式之装饰者模式的学习

发表于 2018-11-10 | 分类于 Java
字数统计: 1.3k | 阅读时长 ≈ 4

装饰者模式

1.介绍

  • 装饰者也称包装模式，是结构型设计模式之一，其使用一种对客户端透明的方式来动态地扩展对象的功能，它是继承关系的一种替代方案。
  • 动态地给一个对象添加额外的职责。

2.使用场景

  • 在不影响其他对象的前提下，以动态、透明的方式添加对象的功能
  • 当不能使用继承的方式对系统进行扩展或者采用继承不利于系统扩展和维护的时候使用
  • 不能采用继承的情况主要有系统中存在大量独立的扩展，为支持每一种扩展或者扩展之间的组合将产生大量的子类，使得子类数目呈爆炸性增长；类已定义为不能被继承（譬如final类），下文继续讲解。

3.UML图

4.组成

4.1 Component

  • Component是基类，通常是一个抽象类或者是接口。
  • 具体构件和抽象装饰类继承于它，其声明了在具体构件中实现的方法。
  • 它的引入可以使客户端以一致的方法处理未被装饰的对象以及装饰后的对象，实现客户端的透明操作。

4.2 ConcreteComponent

  • ConcreteComponent是具体构件，实现了抽象构件的方法，充当被装饰者的角色，装饰器可以给它添加额外的职责。

4.3 Decorator

  • 它是抽象控件的子类，用于给具体构件增加职责，但是具体职责在其子类中实现。它维护一个指向抽象构件对象的引用，通过该引用可以调用装饰之前构件对象的方法，并通过其子类扩展该方法，以达到装饰的目的。不是必须存在的。

4.4 ConcreteDecorator

  • 它是抽象装饰类的子类，负责向构件添加新的职责。 每一个具体装饰类都定义了一些新的行为，它可以调用在抽象装饰类中定义的方法，并可以增加新的方法用以扩充对象的行为。

4.注意事项

  1. 尽量保持装饰类的接口与被装饰类的接口相同，这样，对于客户端而言，无论是装饰之前的对象还是装饰之后的对象都可以一致对待。
  2. 尽量保持具体装饰类是一个“轻”类，也就是说不要把太多的行为放在具体构件类中，我们可以通过装饰类对其进行扩展。
  3. 如果只有一个具体构件类，那么抽象装饰类可以作为该具体构件类的直接子类。
  4. 抽象装饰者类不是一定要有的，除非是有多个装饰类，需要定义一套装饰的规范。
  5. 装饰者类保存的构件的引用，如果可以，尽量存父类抽象构件的引用，这样便可以达到多态的效果，可以同时装饰不同的具体构件。

5.装饰者模式的应用

5.1 Java中的IO

以字节流举例

  • InputStream和OutputStream：它们是我们熟知的两个父类，它们便是装饰者模式下的Component类。
  • FileInputStream等子类：它们便是装饰者模式下的ConcreteComponen，具体实现父类的行为。
  • FilterInputStream：它们是我们熟知的FileInputStream的子类，它便是一个抽象装饰者。
  • BufferedInputStream和DataInputStream：他们是FilterInputStrem的子类，它们便是具体的装饰者了。

6.为什么使用装饰者不使用继承

从装饰者的实现过程中看，如果使用继承完全也可以实现，为什么会出现装饰者模式，为什么有些地方用它替代继承。

  • 如果有final类，则无法继承了。

  • 以应用了装饰者模式的java的IO为例，假如我们自己要实现一套IO系统，以读操作

    • 我们需要有一个InputClass顶层父类，提供一套规范去以便不同的子类去重写读取不同的数据。

    • Input的子类假如有TextInput，MediaInput等…

    • 此时模仿Java的IO系统，为了提升效率，引入缓冲类， 对读方法进行复写，如果使用继承，则

    • 由继承体系可以看出，如果扩展一个功能，则会对应生成两个子类， 扩展性非常不好，也难维护，且会生成多个子类，此时，装饰者模式便起到了用处。

    • 所有的缓冲子类使用的技术都是相同的，没必要都给定义一个子类。现在可以直接单独定义一个缓冲类继承自InputClass类。就是装饰者的角色。
